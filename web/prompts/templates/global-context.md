# Global Project Context

## Project

AI-First Engineering Portfolio Platform

---

## Purpose

This project is not a traditional portfolio website.

It is a long-term software engineering platform designed to demonstrate production-grade engineering practices through interactive software rather than static presentation.

The platform itself is the primary proof of engineering ability.

---

## Vision

Build a maintainable, extensible, AI-native platform that continuously evolves throughout the engineer's career.

Every future AI project should integrate into this platform without requiring architectural redesign.

---

## Target Audience

Primary

- Recruiters
- Hiring Managers
- Senior Software Engineers
- Startup Founders
- Technical Interviewers

Secondary

- Open Source Contributors
- Fellow Developers

---

## Engineering Philosophy

- Simplicity over unnecessary complexity
- Learning before implementation
- Documentation before coding
- AI assists engineering; humans make decisions
- Every architectural decision should be explainable
- Prefer maintainability over cleverness
- Build for long-term extensibility
- Performance is a feature
- Accessibility is mandatory
- Security by default

## Repository Rules

- Feature-first architecture
- Consistent naming conventions
- Small logical commits
- Conventional Commits
- Documentation updated alongside implementation
- No dead code
- No duplicated logic

---

## Documentation Priority

When documentation conflicts with implementation, update the implementation only after updating documentation.

Documentation is treated as a first-class engineering artifact.

---

## Coding Standards

Always prefer

- readability
- maintainability
- modularity
- type safety
- accessibility
- performance

Avoid

- unnecessary abstractions
- premature optimization
- deeply nested components
- magic numbers
- duplicated logic

---

## AI Behavior

When generating code:

- Explain architectural reasoning.
- Follow existing project conventions.
- Do not introduce new libraries without justification.
- Prefer scalable solutions.
- Keep components focused and reusable.
- Document major design decisions.

---

## Success Criteria

Every contribution should improve at least one of:

- maintainability
- scalability
- developer experience
- user experience
- performance
- accessibility
- documentation quality
